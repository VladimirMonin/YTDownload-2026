# YouTube выгрузка в 2026: Playwright + yt-dlp + FFmpeg

> Цель: **стабильно** скачивать видео/аудио (в т.ч. плейлистами), **сливать дорожки** в один файл, и **собирать метаданные** (описание, длительность, обложка, субтитры на разных языках и т.п.) — с учётом **прокси/региональных ограничений**.

---

## 0) Главная идея (как оно работает) 🧩

### Роли инструментов
- **yt-dlp** — «комбайн» для YouTube: извлекает ссылки на потоки, метаданные, субтитры, обложки, комментарии (частично), качает плейлистами, умеет писать `info.json`.
- **FFmpeg** — «склейщик/мультиплексор»: когда YouTube отдаёт видео и аудио раздельно — он **собирает в один контейнер** (часто без перекодирования, очень быстро).
- **Playwright** — «браузерный ключ»: нужен не для скачивания, а для **логина/куков** (и иногда для ручного прохождения бото‑проверок). Далее эти куки использует yt-dlp.

### Почему так (и почему это устойчиво)
- YouTube постоянно меняет внутреннюю выдачу. **yt-dlp** обычно чинят быстрее всех.
- Чисто «питоновские» библиотеки, которые сами парсят YouTube, **чаще ломаются**.
- FFmpeg — стандарт де‑факто для контейнеров/дорожек.

## 1) Что нужно установить (коротко, без боли) 🧰

### 1.1 yt-dlp
- Python пакет: `pip install -U yt-dlp`

### 1.2 FFmpeg
Нужен **обязательно**, если хочешь:
- объединять `bestvideo + bestaudio` в один файл,
- конвертировать/встраивать субтитры,
- конвертировать обложки,
- извлекать аудио в mp3/aac и т.д.

**Windows** (рекомендация): поставить FFmpeg так, чтобы команда `ffmpeg -version` работала в терминале.

### 1.3 Playwright (опционально)
Нужен, если:
- часто выскакивает “Sign in to confirm you’re not a bot”,
- нужен доступ к 18+/приватным роликам,
- нужен стабильный процесс логина через браузер.

Установка:
- `pip install playwright`
- `playwright install`

## 2) Прокси: что выбрать и как подключить 🛰️

### 2.1 Что «лучше» для YouTube
**Лучшее для скачивания** (особенно плейлистов) — это:
- ✅ **стабильный IP** (не ротирующийся каждые 2–5 минут)
- ✅ адекватная скорость и отсутствие микро‑обрывов

**Ротация IP** чаще полезна для массового **сбора метаданных**, но может мешать:
- ломает куки/сессии,
- повышает шанс капч,
- рвёт докачку больших файлов.

### 2.2 Типы прокси (практический рейтинг)
1) 🥇 **SOCKS5 (локальный туннель)** через Clash/V2Ray/WireGuard/Outline и т.п.
   - удобно, универсально, стабильно
2) 🥈 **HTTP/HTTPS proxy**
   - ок, если socks нет
3) 🥉 публичные/бесплатные прокси
   - почти всегда страдания (скорость/обрывы/баны)

### 2.3 Как прокси подключается к yt-dlp
Есть два подхода:

**A) Системный (рекомендуется)**
- Включаешь VPN/туннель на уровне ОС.
- yt-dlp просто работает «как обычно», без прокси‑настроек в коде.
- Плюсы: меньше возни, одинаковая среда для Playwright и yt-dlp.

**B) Явно в yt-dlp**
- Передаёшь `proxy` как строку вида:
  - `socks5://user:pass@host:port`
  - `http://user:pass@host:port`
- Плюсы: можно включать/выключать для конкретных задач.
- Минусы: надо следить, чтобы Playwright и yt-dlp «видели мир» одинаково.

### 2.4 Ключевой нюанс: Playwright + yt-dlp должны быть в одной «географии»
Если ты:
- логинишься в YouTube в браузере **через один IP**,
- а скачиваешь через yt-dlp **через другой**,

…YouTube может начать «коситься»: просить подтверждения, сбрасывать куки или ограничивать выдачу.

✅ Правило: **один стабильный выход в сеть** для всех этапов.

## 3) Нужен ли аккаунт? (и как сделать “тех‑акк”) 👤

### 3.1 Когда аккаунт реально помогает
- Часто вылезает “confirm you’re not a bot”
- Нужны ролики 18+
- Доступ к приватным/доступ по ссылке
- Нужны плейлисты “для себя” (Watch later / Liked)

### 3.2 Почему лучше отдельный аккаунт
- меньше рисков для основного Google аккаунта,
- можно держать отдельный браузерный профиль + отдельные куки,
- проще дисциплина: «этот аккаунт только для скачиваний».

### 3.3 Главная дисциплина при использовании аккаунта
- ❗ **не менять IP на лету**
- ❗ **не качать слишком агрессивно** (много параллели)
- ✅ лучше медленнее, но стабильно

## 4) Куки: 2 рабочих стратегии (без лишней магии) 🍪

### Стратегия №1 (самая простая): `cookies-from-browser`
Если yt-dlp видит установленный браузер (Chrome/Firefox), обычно можно:
- забрать куки прямо из профиля браузера,
- без Playwright.

Плюсы:
- минимум кода,
- меньше движущихся частей.

Минусы:
- иногда сложнее, если нужен отдельный изолированный профиль или portable‑режим.

### Стратегия №2: Playwright «открывает браузер → ты логинишься → сохраняем cookies»
Плюсы:
- полностью контролируемый “тех‑профиль”,
- можно руками пройти проверки.

Минусы:
- чуть больше инфраструктуры.

💡 Совет: начни со Стратегии №1. Если упираешься — подключай Playwright.

## 5) Что именно может собирать yt-dlp (метаданные и файлы) 📦

### 5.1 Метаданные (внутри `info` / `info.json`)
Обычно доступны:
- **title** (заголовок)
- **id** (YouTube id)
- **uploader / channel / channel_id**
- **upload_date**
- **duration** (секунды)
- **description**
- **tags**
- **thumbnails** (список вариантов обложек)
- **formats** (список доступных потоков: кодек/размер/битрейт/контейнер)
- **subtitles / automatic_captions** (какие языки есть)
- **chapters** (если есть)

Практика:
- почти всегда можно сначала **только извлечь метаданные** (без скачивания),
- и уже потом решать: что качать, какие дорожки, какие языки.

### 5.2 Что он может сохранить отдельными файлами
- 🧾 `info.json` (полный слепок метаданных)
- 📝 описание в `.description`
- 🖼️ обложка (thumbnail)
- 🧷 субтитры (ручные) + авто‑сабы
- 🗂️ главы/таймкоды (косвенно через метаданные, иногда — отдельные форматы)
- 💬 комментарии (частично/не всегда стабильно; зависит от ролика и ограничений)

## 6) Субтитры: языки, форматы, “авто” и встраивание 🗣️

### 6.1 Источники субтитров
- **ручные (human)** — если автор/команда загрузили
- **автоматические (auto)** — генерируются YouTube

### 6.2 Практические советы
- Если тебе нужен русский/английский/казахский:
  - сначала скачай **список доступных языков** (без загрузки видео)
  - затем запрашивай нужное

- С автосабами нюанс:
  - часто доступны в VTT,
  - иногда качество заметно хуже, чем ручные.

### 6.3 Форматы
Обычно встречаются:
- `vtt` (часто у YouTube)
- `srt` (удобно для людей)

Рекомендация:
- скачивай `vtt`, потом при необходимости конвертируй в `srt` через FFmpeg или средствами yt-dlp.

### 6.4 Встраивание субтитров в видео
- “встроить” (embed) — значит положить текстовую дорожку внутрь контейнера MKV/MP4.
- MP4 иногда капризнее, MKV чаще «всё ест».

Если курс/материалы — часто лучше хранить субтитры отдельными файлами: проще индексировать, проще править.

---

## 7) Обложки (thumbnail) и превью 🖼️

Что важно:
- У ролика может быть несколько обложек разных размеров.
- yt-dlp обычно умеет выкачать “best” обложку.

Практика для индекса:
- сохранять thumbnail рядом с видео,
- а в `info.json` оставлять ссылку + параметры.

Доп. нюанс:
- иногда обложки приходят в `webp` — может понадобиться конвертация (если твой пайплайн не любит webp).

---

## 8) Плейлисты: контроль, стабильность, «не всё сразу» 📚

### 8.1 Что важно при скачивании плейлистов
- Умей ограничивать диапазон:
  - «скачай только 1–20», «скачай 5,7,9»
- Умей продолжать докачку:
  - если оборвалось, должно продолжиться

### 8.2 Параметры поведения (по смыслу, без кода)
Рекомендуемые включения:
- ✅ продолжение докачки
- ✅ повторные попытки
- ✅ игнорировать “битые” элементы плейлиста (чтобы не падал весь процесс)
- ✅ умеренная параллель

Идеальная стратегия:
1) Сначала «плоско» получить список элементов (быстро)
2) Потом качать по одному/маленькими пачками

## 9) «Склейка» аудио+видео: что реально происходит 🔧

### 9.1 Почему дорожки раздельные
YouTube часто отдаёт:
- **видео** без аудио (высокое качество)
- **аудио** отдельно (bestaudio)

### 9.2 Что делает FFmpeg
- “mux” (мультиплексирование): **перепаковывает** потоки в контейнер (MP4/MKV)
- обычно без перекодирования (`copy`) → быстро и без потерь

### 9.3 Выбор контейнера
- **MP4** — совместимость (телевизоры/телефоны)
- **MKV** — удобство (сабы/дорожки/метаданные, менее капризный)

Если в курсе важна «всё работает везде» — MP4.
Если важна «всё хранится аккуратно» — MKV.

---

## 10) Комментарии: можно, но это не “железный” функционал 💬

yt-dlp может пытаться выгрузить комментарии, но:
- иногда работает отлично,
- иногда упирается в ограничения/логин/скорость,
- иногда YouTube меняет API и это ломается.

Практическая рекомендация:
- если комментарии нужны **обязательно**, делай это отдельным шагом и будь готов к нестабильности.
- для курса чаще важнее **описание + таймкоды + сабы**, чем комментарии.

## 11) Режимы работы (под твой “курс/индексацию”) 🧠

### Режим A: «Сначала метаданные → потом скачивание» (лучший для системности)
1) Извлечь `info` без скачивания
2) Сохранить `info.json`
3) Проверить:
   - длительность
   - доступные языки субтитров
   - наличие глав
   - доступные форматы (AV1/VP9/H.264, аудио codecs)
4) Скачать по выбранным правилам

Плюсы:
- меньше сюрпризов,
- проще строить индекс и пайплайн.

### Режим B: «Сразу качаем best» (быстро, но меньше контроля)
Подходит, когда:
- это не массовая выгрузка,
- не важно тонко настраивать форматы.

## 12) Практические настройки “чтобы было стабильно” 🧯

### 12.1 Сеть и ретраи
Если часто обрывается:
- увеличь количество повторов
- уменьшай параллельность
- добавь небольшой “сон” между элементами плейлиста

### 12.2 Ограничение скорости
Иногда YouTube «душит» скорость при слишком быстром скачивании.
Тогда помогает:
- ограничить rate,
- качать без агрессивной параллели,
- стабильный IP.

### 12.3 Отдельная папка и шаблоны имён
Для курсов/архивов важны:
- понятные имена файлов
- индексация по номеру в плейлисте
- сохранение рядом `info.json`, `thumbnail`, `subs`

Пример структуры (как идея):
- `CourseName/`
  - `001 - Title/`
    - `video.mp4`
    - `info.json`
    - `thumb.webp`
    - `ru.vtt`
    - `en.vtt`

## 13) Безопасность и гигиена (важно) 🔒

- Куки = доступ к твоему аккаунту. Храни `cookies.txt` как пароль.
- Не коммить в git.
- Если делаешь «тех‑акк» — не держи на нём ничего личного.
- IP и аккаунт должны быть предсказуемыми: меньше “подозрительной активности”.
